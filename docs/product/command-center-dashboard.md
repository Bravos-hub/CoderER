# CodeER Command Center Dashboard

## Objective

The dashboard must feel like a real software incident-response platform rather than a chatbot with a sidebar. It should make repository health, active incidents, agent activity, evidence, proposed changes, and verification status visible at a glance.

## Primary navigation

```text
Command Center
Active Incidents
Repositories
Recovery Sessions
Verification
Pull Requests
Case History
Settings
```

## Command Center overview

### Header

```text
CodeER Command Center
3 repositories monitored
1 critical incident
2 recoveries in progress
```

### Summary cards

- Repositories monitored
- Active incidents
- Recoveries in progress
- Verified recoveries

Each card should link to a filtered view.

## Repository health model

Repository health is scored from 0 to 100.

|  Score | Status     |
| -----: | ---------- |
| 90–100 | Stable     |
|  75–89 | Recovering |
|  50–74 | Degraded   |
|  25–49 | Critical   |
|   0–24 | Unstable   |

### Health dimensions

- Build health
- Test health
- Security
- Deployment readiness
- Dependency health
- API consistency
- Frontend functionality
- Documentation readiness

The score must never appear arbitrary. Every dimension must expose the checks, failures, and evidence used to calculate it.

## Incident severity

### SEV-1 — Critical

- Production unavailable
- Security exposure
- Data corruption risk
- Authentication completely broken

### SEV-2 — High

- Core functionality unavailable
- Deployment blocked
- Major integration failure

### SEV-3 — Medium

- Partial feature failure
- Test or quality degradation
- Recoverable configuration issue

### SEV-4 — Low

- Maintainability issue
- Minor regression
- Documentation or cleanup task

## Active Incident workspace

Use a three-column layout on large screens.

### Left column — Incident evidence

Show:

- Original error
- Failing command
- Workflow or deployment source
- Relevant logs
- Environment details
- Suspected files
- Related commits
- Reproduction status

### Center column — Response-team activity

Example timeline:

```text
10:42 Repository Mapper indexed 482 files
10:43 Investigator reproduced the build failure
10:45 Contract Inspector found an API path mismatch
10:47 Repair Agent prepared a treatment plan
10:49 Security Reviewer checked the proposed patch
```

Every event should include:

- Agent name
- Timestamp
- Action
- Evidence link
- Outcome

### Right column — Recovery status

Show:

- Severity
- Current stage
- Confidence
- Files affected
- Estimated blast radius
- Required verification checks
- Approval status
- Stop or cancel controls

## Recovery-plan screen

Required sections:

```text
Root cause
Evidence
Affected files
Proposed changes
Potential risk
Validation plan
Rollback plan
```

Primary actions:

- Approve procedure
- Request revision
- Reject treatment

No patch should be applied until the user can see the intended files and verification plan.

## Code-diff review

The review screen must provide:

- Side-by-side and unified diff views
- Changed-file navigation
- Added, modified, and removed file counts
- Agent explanation for each change
- Warnings for configuration, security, migration, and dependency changes
- Ability to exclude a proposed file before approval

## Verification screen

Required checks:

- Original failure removed
- Dependency installation passed
- Lint passed
- Type checking passed
- Unit tests passed
- Integration tests passed
- Production build passed
- Critical browser journey passed
- No unexpected file changes
- Security policy checks passed

Each check should show:

- Status
- Command
- Duration
- Relevant output
- Evidence artifact

## Pull-request preview

Show:

- Pull-request title
- Root-cause summary
- Changed files
- Before-and-after evidence
- Verification report
- Known limitations
- Rollback instructions
- User approval controls

## Interaction principles

- Show progress without hiding uncertainty.
- Make destructive or irreversible actions explicit.
- Keep technical commands and logs copyable.
- Use status labels in addition to color.
- Allow users to pause or cancel a recovery.
- Never report a repair as verified until all required checks complete.
- Preserve a full audit trail for each recovery session.
